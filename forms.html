Forms

We use forms to give customers the ability provide us with information.

Forms matter because the failure of form results in failure demand or a loss sale. We don't like this stuff.

So let's get forms right.

The 4 c's of form design

  • Clear questions which have a clear and unambiguous answer.
  • Concise wording without fluff or jargon
  • Clever so that we offer help if a customer makes a mistake
  • Co-operative inputs so that even if the user strays from the "normal use case" their data can still be used.

Section heading

Use the field set to represent a set of form controls e.g ADDRESS DETAILS or Payments. The headings separate sections on the page and make it easy for the user to navigate the form.

Heading should clearly summarise the content of the section.

Headings should be distinct from each other and unambiguous.

Example

<h2 class="Title-2">ADDRESS DETAILS</h2>

Sub section heading

Use a subheading when there is multiple groups of form controls within a section e.g ADDRESS section includes sub-section of Home address and Postal address.

Use a sub section heading when there are multiple inputs to answer. For example radio buttons or check boxes. The sub section heading contains the "question text" in these situations.

Labelling

A good label significantly helps form conversion by adding clarity without overwhelming users.

The purpose of the label is to indicate what data is required in the field. All input fields are required to have labels.

Labels should be clear and easy to understand. If label is ambiguous users will either guess an answer causing problems down the track or leave the form altogether.

Usage for Text Inputs

  • Labels are to be above the field.
  • They are to be aligned to the left of the input.

Usage for Radio Buttons

  • Aligned to the right of the input
  • Visually grouped with the input
  • Visually separated from other radio buttons.

Example

<label class="control-label">PHONE NUMBER</label>

CSS

.control-label {
display: block;
margin-bottom: 10px;
color: #fff;
font-size: 14px;
font-weight: 700;
text-transform: uppercase;
};

Text input

Use a text input field when:

  • A string of characters
  • A single line of content

Avoid

Field length that goes over 60 characters as this creates poor readability. In an extreme case 90 Characters is an absolute maximum.

Radio buttons

Radio Buttons are used when a choice is required between two or more options. Multiple options can be selected when radio buttons are used.

User must select an answer (if answer not mandatory then use checkbox)

Each choice should be clear and distinct. We want to avoid users humming and ahhing between options

Usage:

Each group of inputs has a sub section heading which contains the “Question text”

Each radio button is to have a label which contains the answer

Preference is to use one word answers.

Example

<input id="male">Male></input>

Checkbox

Used when a choice needs to be made between one or more options

Multiple options can be selected

Used when an answer is optional

Each choice should be clear and distinct. We want to avoid users guessing between options

Usage for single checkboxes

Single checkboxes are used for _optional_ agreement to participate in marketing programs or for optional

Wherever possible single checkboxes should be _avoided for mandatory acceptance of terms and conditions or legal disclaimers_. It is better to have mandatory acceptance acknowledged by the act of submitting the form.

Check box is left aligned to label Elements are to be grouped visually so connection is strong

Usage for multiple checkboxes

Each group of inputs has a sub section heading which contains the “Question text”

Each radio button is to have a label which contains the answer. Preference is to use one word answers.

Ensure clear separation between each "input and label" pair.

Use visual grouping of label and input though background colour and border

This avoids confusions over which radio button relates to which label

Alerts & Validation

Alerts are to following the following pattern. Advise the user what has happened and - if applicable - what the next steps are.

Success

Advise the customer their input is valid

Success on valid input - client side

We like to make customers feel good by showing their input is valid in context and as soon as possible.

We do this by putting a tick next to the input field on blur.

When they reenter the field the tick remains until the input becomes invalid again. They are then shown the alter message

Success on form submission - server side

If success message confirms submission of the form always advise customer how long any processing times or activation times. For example let them know how long until they will receive a claim or how long until we will call them back with further questions.

Always remove doubt or uncertainty - if there is nothing else they have to do then tell them that. They will enjoy the confirmation.

Alert

We like to alert customers when they have entered information that is invalid or when we need to get their attention.

The best alerts let the customer what has gone wrong (e.g. a phone number missing numbers) then we tell them what to do to fix.

We avoid blaming the customer and we always aim for clarity of information.

Don’t leave them guessing about how to fix the error. tell them what to do next.

To alter customers we utilise orange and yellow where possible. It is friendly and still gets their attention.

Example:

<div class="tip south"></div>
Please enter a street address

Supporting information

Validation rules

If an input has validation rules then we always tell the customer what these are.

Explain character limits or invalid characters. This information should always be visible using simple clear wording.

Guidance

Supporting information can also be used to help explain complex terms. If a single line can be used close to the field then the information can be visible.

However sometimes we need to use images, tables or larger amounts of text to help the customer .In this case it is best to hide the information and reveal it to the customer when ready. This avoids overwhelming customers with too much content or distracting them needlessly.

To activate supporting information utilise a info icon or a "help information" link.

Example:

<p class="text-small">Disclaimer</p>
